# QuickPresence
#### Video Demo: <URL HERE>
#### Description:

QuickPresence is a Ruby on Rails web application for creating temporary attendance lists that can be shared through a public link or QR Code. It was built as my CS50x final project to solve a practical problem I have seen in classes, meetings, workshops, and small events: organizers often need to collect attendance quickly, but they do not need a full survey platform or a system that requires every participant to create an account. QuickPresence lets an organizer create a list, choose the information that should be collected, define the time window in which responses are accepted, and then share a simple public form with participants.

The application has two main kinds of users. The organizer is authenticated and can create, edit, close, review, and export attendance lists. The participant does not need to sign in. They open a public URL or scan a QR Code, fill in the fields configured by the organizer, and submit their attendance while the list is open. After submission, participants are redirected to a confirmation page instead of seeing a blank form again, which avoids confusion and reduces accidental duplicate submissions.

Each attendance list can have up to five custom text fields. This limit is intentional: the project is meant to be quick and focused, not a general form builder. A teacher might collect name, student ID, and class; an event organizer might collect name and company. The submission timestamp is generated by the server and does not count as one of the custom fields. Lists also have start and end times, and each list stores its own time zone so that the organizer's intended schedule remains clear even if the server runs in a different zone.

Responses are retained for a limited period. After an attendance list ends, submitted responses remain available for review and export for 48 hours. A recurring background job then removes expired responses and answers. This design reflects the privacy goal of the project: attendance data should be kept only as long as it is useful for the organizer's immediate purpose.

## Main Features

- Username-based organizer registration and login with Devise.
- CRUD for attendance lists owned by the current user.
- Up to five custom fields per attendance list.
- Public attendance forms available at tokenized URLs.
- QR Code generation for public links.
- List availability controlled by start and end time.
- Server-generated response timestamps.
- CSV and XLSX exports.
- Printable QR Code PDF generation.
- Public confirmation page after successful attendance submission.
- Privacy notice page and first-access privacy modal.
- English and Brazilian Portuguese translations through Rails I18n.
- Automatic cleanup jobs with Solid Queue.
- SQLite-backed production setup for a small single-VM deployment.

## Files and Directories

The Rails models in `app/models` define the core data relationships. `User` represents an organizer account. `AttendanceList` belongs to a user and owns fields and responses. `AttendanceField` represents each custom field configured for a list. `AttendanceResponse` stores a participant submission, including the timestamp, IP address, and user agent. `AttendanceAnswer` stores each answer linked to both a response and a field.

The controllers in `app/controllers` handle the user workflows. `AttendanceListsController` manages the authenticated organizer experience: listing, creating, editing, closing, exporting, and reviewing lists. `PublicAttendanceController` handles the public form, closed-list state, response creation, and final confirmation page. `PrivacyController` renders the privacy notice. `ApplicationController` centralizes locale selection and Devise parameter handling.

The services in `app/services` keep larger operations out of controllers. `AttendanceListExport` builds export rows shared by export formats. `AttendanceListXlsx` creates a minimal XLSX file. `AttendanceListQrCodePdf` renders a printable PDF with the public QR Code. `AttendanceResponsesPurger` removes expired response data. `BrowserLocale` resolves the preferred locale from the browser's `Accept-Language` header.

The views in `app/views` are server-rendered ERB templates. The organizer views live under `app/views/attendance_lists`. The participant-facing public form and confirmation screens live under `app/views/public_attendance`. The shared layout in `app/views/layouts/application.html.erb` includes the navigation, flash messages, Bootstrap assets, and privacy modal.

The JavaScript in `app/javascript/application.js` is intentionally small. It detects the organizer's browser time zone when creating a list, supports copying public links to the clipboard, and records when the privacy notice has been acknowledged. I chose to avoid a heavy front-end framework because Rails-rendered views are enough for the project and keep the code easier to understand.

The recurring jobs in `app/jobs` and `config/recurring.yml` run cleanup work in the background. `PurgeExpiredAttendanceResponsesJob` deletes expired attendance responses. `DeleteInactiveUsersJob` removes accounts that meet the inactivity deletion criteria. Solid Queue stores the queue data in SQLite, matching the single-VM deployment approach.

The files in `config/locales` contain all user-facing text. I kept English as the default locale and added Brazilian Portuguese translations. The test suite includes an I18n catalog test to ensure the Portuguese files keep parity with English translation keys.

The `docs` directory contains supporting documentation that is more detailed than this README. `docs/quick_presence_escopo_cs50.md` records the original project scope and planning notes. `docs/project-guidelines.md` documents engineering rules, architecture decisions, and SOLID principles used in the project. `docs/development-and-operations.md` preserves the previous technical README with development and deployment notes. Some operational files that could reveal production details are intentionally ignored by Git.

## Design Choices

I chose Ruby on Rails because it provides a productive full-stack structure: routing, controllers, views, models, migrations, background jobs, security helpers, and testing tools all work together. That allowed me to focus on the attendance workflow instead of wiring together many separate technologies.

SQLite was chosen because the current deployment target is a small single VM. It keeps production simple: there is no separate database server to administer, and the database files can live in a persistent storage directory mounted into the container. This would not be the right choice for a multi-server or high-write workload, but it fits the current scope of QuickPresence.

The public form uses a random token instead of a numeric attendance list ID. This makes shared links harder to guess and keeps private organizer routes separate from public participant routes. Authenticated actions always load lists through `current_user.attendance_lists`, so one organizer cannot manage another organizer's lists.

I also chose to implement a dedicated confirmation page after public submission. The earlier behavior redirected participants back to the form with a success message, but that could make people think they needed to fill it out again. A final confirmation page is clearer and more respectful of the user's time.

For privacy, the app currently uses only essential cookies and browser storage: login/session handling, CSRF protection, static asset caching, and a local flag remembering that the privacy notice was acknowledged. It does not use analytics cookies, advertising pixels, behavioral profiling, or marketing trackers. If optional tracking is ever added, it must be disabled by default and loaded only after clear consent.

## How to Run

Install dependencies:

```bash
bundle install
```

Prepare the database:

```bash
bin/rails db:prepare
```

Run the development server:

```bash
bin/dev
```

Run the tests:

```bash
bin/rails test
```

## Production Summary

The production setup uses Docker Compose with a prebuilt Rails image, Caddy for HTTPS, and SQLite files stored in a persistent host directory. The production deployment details that expose operational information are not all included in the public repository, but the general approach is documented in `docs/development-and-operations.md` and `docs/production-maintenance.md`.
